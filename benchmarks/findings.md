# Dashboard Studio Custom Visualization — Performance Findings

**Author:** Maintainer of [splunk-icon-library](https://github.com/lyderhansen/splunk-icon-library) (a public Material Symbols icon viz for Dashboard Studio)
**Date:** 2026-05-27 (updated; original 2026-05-24)
**Apps under test:** `icon_library` 1.3.3 (initial public release) and 1.5.4 (post optimization pass), **plus** `Semicircle Donut Chart` — an unrelated third-party custom viz from Splunkbase ([app 4378](https://splunkbase.splunk.com/app/4378)) — at 250 panels
**Splunk Enterprise build under test:** static asset hash `f798d4d49089.276` / `.298` (localhost dev instance)
**Browsers:** Chrome and Safari, both on macOS

---

## TL;DR

Dashboards composed of many Custom Visualization (iframe-isolated) panels exhibit a **per-panel framework re-fetch pattern** that does not occur with native visualizations. The pattern is **independently reproduced across two unrelated, community-built custom-viz apps from Splunkbase** (our own `icon_library` and the unrelated `Semicircle Donut Chart` — Splunkbase [app 4378](https://splunkbase.splunk.com/app/4378)), which rules out anything app-specific and points to the Custom Visualization runtime itself:

- ~10 framework chunks (bootstrap CSS, backbone.js, require-css, custom-viz iframe scripts, Proxima Nova fonts) **each fetched once per panel** — 256× on icon_library, 250× on semicircle_donut.
- **3,426 (icon_library) / 3,870 (semicircle_donut) total HTTP requests** for a ~250-panel dashboard, vs **277 requests** for an equivalent 250-panel dashboard built from native `splunk.singlevalue` panels.
- **All framework chunks finish loading at +70 s (semicircle_donut) / +84 s (icon_library)** — vs **+34 s** for the native equivalent.
- Transfer ballooning was browser-dependent: Chrome served most repeats from the HTTP cache (~12 MB on the wire for both custom-viz apps); Safari did not (99.5 MB on icon_library).
- **Page `onLoad` timing is misleading and varies wildly between apps** (1.2 s for semicircle_donut vs 76.6 s for icon_library) because some custom vizes happen to block the parent's onLoad event and others don't. **The reliable metric is the time at which the last framework chunk finishes loading**, which is consistent across apps (~70–84 s) and reflects when iframes actually finish their per-panel bootstrap.

App-side optimization on icon_library eliminated everything we could reach (data-payload trimming, render skipping, font loading, DPR clamping, observer narrowing). The Semicircle Donut Chart app, written and shipped by a different developer, hits the same wall — confirming the bottleneck is the per-iframe framework fetches that the app code cannot influence.

This document is intended as a bug / improvement report for the Dashboard Studio team, with the goal of getting the iframe framework chunks served from a shared cache (or, ideally, hoisted out of the iframe boundary entirely) so that custom-viz dashboards scale comparably to native-viz dashboards.

| Last framework-chunk loaded by | **+34 s** | **+84 s** | n/a* | **+70 s** |
| Page `onLoad` event | 33.5 s | 76.6 s | 97.4 s | **1.2 s ⚠** |
| Total bytes transferred | 0.4 MB | 12.1 MB | 99.5 MB | **11.4 MB** |
| Distinct URLs | 23 | 169 | 52 | 420 |

\* not parsed for the Safari capture; the Chrome icon_library and semicircle_donut numbers already establish the pattern.

⚠ **The Page `onLoad` event is unreliable for measuring custom-viz dashboard load time** — for semicircle_donut it fires at 1.2 s even though 98 % of the dashboard's requests happen *after* onLoad. Why depends on whether the viz happens to block the parent's onLoad event (icon_library does — likely via its font-loading promise — semicircle_donut does not). **The reliable number is "last framework-chunk loaded by"**: that's when the last iframe finishes its bootstrap, and it's consistent across apps (+70–84 s for the custom-viz dashboards, +34 s for the native baseline).

Normalized to per-panel cost across the two custom-viz dashboards:

| Per-panel | icon_library (Chrome) | semicircle_donut (Chrome) |
|---|---|---|
| Requests per panel | ~12.8 | ~15.5 |
| Bytes per panel | ~45 KB | ~46 KB |
| Framework-loaded delta vs native | +50 s / panel-batch | +36 s / panel-batch |

For comparison, the native dashboard runs ~1.1 requests and ~1.6 KB per panel — **between 10× and 30× cheaper per panel** depending on which custom viz is involved.

## The smoking gun: repeated framework chunks

Top URLs by request count on the custom-viz dashboards (Chrome capture, `icon_1.3.3.har`):

```
 256 × .../app/splunk-dashboard-studio/build/custom-viz-iframe-scripts/legacy.js
 256 × .../build/css/bootstrap-enterprise.css
 256 × .../build/custom_viz/dark/minimal_exposed_modules.js
 256 × .../js/require/backbone.js
 256 × .../js/contrib/backbone.js
 256 × .../js/splunkjs/contrib/require-css/css.js
 256 × .../js/splunkjs/contrib/require-css/normalize.js
 256 × .../fonts/proxima-bold-webfont.woff
 256 × .../fonts/proxima-regular-webfont.woff
 256 × .../fonts/proxima-semibold-webfont.woff
```

The Safari capture shows the same pattern with the request count tracking the panel count (268× rather than 256× — the extra 12 are demo blocks elsewhere in the dashboard):

```
 268 × .../app/splunk-dashboard-studio/build/custom-viz-iframe-scripts/legacy.js
 268 × .../build/css/bootstrap-enterprise.css
 268 × .../build/custom_viz/dark/minimal_exposed_modules.js
 268 × .../js/require/backbone.js
 268 × .../js/contrib/backbone.js
 268 × .../js/splunkjs/contrib/require-css/css.js
```

These chunks are byte-identical across panels — they are the Dashboard Studio framework's own runtime (jQuery contrib, Backbone, require-css, the custom-viz iframe shim, the bootstrap-enterprise theme, and the Proxima Nova font family used for Splunk's own UI chrome). They are unrelated to our app's bundle (`appserver/static/visualizations/icon_library/visualization.js`), which is also re-fetched per iframe but only ships at 13.6 KiB minified.

**Each Custom Visualization panel is its own iframe and re-pays the cost of bootstrapping the Splunk framework as if it were a top-level page.** Chrome's HTTP cache deduplicates the bytes on the wire (12 MB transferred for 3,400+ requests), but the parse / evaluate / module-register cost is paid 256 times even when the response is `304 Not Modified` — and the request count itself starves the connection pool. Safari does not deduplicate cross-iframe nearly as aggressively (99.5 MB on the wire) which is why its `onLoad` is 21 s slower than Chrome's on the same dashboard.

The partial `icon_03.har` capture (cut off mid-load) confirms the same pattern within its window:

```
 231 × http://localhost:8002/static/.../icon_library/visualization.js
 231 × .../js/splunkjs/contrib/require-css/css.js
  43 × .../js/contrib/backbone.js
  21 × .../js/splunkjs/contrib/require-css/normalize.js
```

— so the duplication is happening continuously during load, not as a one-time burst.

## Independent reproduction with a second custom viz

To rule out anything app-specific, we ran the same benchmark with **`Semicircle Donut Chart`** — a community-built Splunkbase custom viz ([app 4378](https://splunkbase.splunk.com/app/4378)), by a different developer, with no shared code with `icon_library` and no perf-pass history. We dropped 250 instances of `semicircle_donut.semicircle_donut` into a dashboard, each backed by its own `| makeresults | eval count=N` data source, and captured the HAR (`semi_circle_donut.har`, 244 MB).

Same pattern, same magnitudes:

```
 250 × .../app/splunk-dashboard-studio/build/custom-viz-iframe-scripts/legacy.js
 250 × .../build/css/bootstrap-enterprise.css
 250 × .../build/custom_viz/dark/minimal_exposed_modules.js
 250 × .../js/splunkjs/contrib/require-css/normalize.js
 250 × .../fonts/proxima-bold-webfont.woff
 250 × .../fonts/proxima-regular-webfont.woff
 250 × .../fonts/proxima-semibold-webfont.woff
 250 × .../fonts/inconsolata-regular.woff
 250 × .../fonts/splunkicons-regular-webfont.woff
 247 × .../js/require/backbone.js
 235 × .../js/splunkjs/contrib/require-css/css.js
 225 × .../js/contrib/backbone.js
```

The minor variation in counts (247, 235, 225 instead of a clean 250) appears to be cache-hit deduplication during the load — the request still happens but Chrome occasionally serves it from in-memory cache without logging a separate entry. The dominant signal — **the framework chunks each fetched once per panel** — is identical.

This is the smoking gun for "this is a runtime cost, not an app cost." Two unrelated Splunkbase apps, written by different developers, with different rendering pipelines and zero shared code, pay the **identical** per-panel iframe bootstrap. No app-side code change can fix it; the fix has to be in the Dashboard Studio framework itself.

## What we ruled out before filing this

We exhausted everything an app author can do from inside the viz before concluding the bottleneck is the framework. The version difference (1.3.3 → 1.5.4) lets us isolate app-side contribution.

App-side perf changes shipped between 1.3.3 and 1.5.4:

| Change | Before | After |
|---|---|---|
| `getInitialDataParams.count` | `10000` (default) | `1` |
| Per-update render | Always re-render canvas | Skip render when `data` and `config` are identity-equal |
| Font load | Wait for `document.fonts.load()` before first paint (blank panel) | Paint with fallback font immediately; re-paint when Material Symbols resolves |
| Font-resolved callback fan-out | All N panels in one tick | Staggered, 8 panels per animation frame |
| `devicePixelRatio` | Native (up to 3 on 5K displays) | Clamped to 2 |
| No-data placeholder `MutationObserver` | `subtree: true`, never disconnects | `subtree: false`, self-disconnects after first successful render |
| Canvas memory | Full DPR backing buffer | Halved on HiDPI |

Net result: 1.5.4 cold-loaded the same 256-panel showcase ~33 s faster than 1.3.3 on the same hardware. That improvement is the app-side ceiling — what remains in the `onLoad` budget is overwhelmingly framework-iframe overhead (the 10 chunks × 256 iframes pattern above).

Things we verified were **not** the bottleneck:

- Material Symbols font load. We embed the font as a base64 `data:` URL inside `visualization.css` (a single 270 KB asset shipped once per iframe), then call `document.fonts.load(...)`. Font load resolves within a few hundred ms even on cold load and is dwarfed by framework startup.
- Our render code. The Canvas 2D path is `<5 ms` for a single icon on commodity hardware, and `count: 1` brought the SPL data payload to negligible size.
- Search execution. The showcase uses `| makeresults` which Splunk dispatches in 50–150 ms per panel. The native baseline uses the same SPL pattern and loads in ~30 % of the time.
- Our app's own bundle size. `visualization.js` is 13.6 KiB minified; the framework chunks dwarf it.

## Why this affects more than our app

Any Splunkbase app that ships a Dashboard Studio custom visualization will hit this wall as soon as a dashboard exceeds ~50 panels. Categories of dashboards that are common in customer environments and that easily clear that bar:

- Status walls (one tile per service / region / customer / asset).
- Datacenter rack views, network topology overlays, IoT fleet status.
- ITSI / SOC mosaics with one icon-and-metric per glass entry.
- Catalog or directory dashboards (which is exactly what our `Showcase` is — 256 icon previews).

In our experience, dashboard authors discovering this for the first time read it as "this app is slow" rather than "Custom Visualizations have a per-panel iframe cost," which we believe disincentivises adoption of Splunkbase custom-viz apps generally.

## Asks for the Splunk team

In rough order of likely tractability and impact:

1. **Hoist the framework chunks out of the iframe boundary.** The custom-viz iframe needs an isolation barrier for the *visualization code itself*, but the Dashboard Studio runtime, Backbone, require-css, bootstrap-enterprise, and the Proxima font family are framework dependencies that could be loaded by the parent document and exposed to child iframes via `parent.<symbol>` rather than re-bootstrapped 256 times.

2. **Or, share via a single hidden iframe.** A common pattern in micro-frontend frameworks: load the framework into one hidden iframe per dashboard, and have visualization iframes inherit from it (`document.write` of a `<base>` plus shared module registry). Bypasses the per-panel parse cost without losing isolation.

3. **Or, at minimum, set far-future `Cache-Control` + `immutable` on the hashed framework chunks** and serve them with `Cross-Origin-Resource-Policy: cross-origin` so cross-iframe HTTP cache hits don't even register as requests. The hashed paths (`@f798d4d49089.276/...`) imply this is already content-addressed, but Safari's 99.5 MB number suggests the cache headers aren't communicating immutability to all browsers.

4. **Document the per-panel cost in the [Custom Visualization Developer Guide](https://help.splunk.com/en/splunk-enterprise/develop-on-splunk/dashboard-studio-developer-guide-for-custom-visualizations).** Right now app authors discover this pattern only by capturing HAR files and counting URLs. A note in the guide that says "Custom Visualizations are one iframe per panel; expect framework chunks to be requested per-panel; avoid >50-panel dashboards or build a multi-record viz" would let authors plan accordingly.

5. **Consider an opt-in "shared-context" mode** for trusted custom vizes that don't need iframe isolation (e.g. ones that pass AppInspect and are signed). The cost/benefit of iframe isolation differs sharply between hostile-third-party plugins and an internal app written by the same team running Splunk.

## What we are doing on our side

We have done what an app author can reasonably do. We are now scoping a "mega-panel" mode for v2.0 of this app — a second viz type that renders N icons in a *single* iframe via internal hit-testing and event dispatch. This eliminates the per-panel iframe cost for users who can express their layout as a grid rather than as N panels, but it is not a generally applicable fix: dashboard authors lose Splunk Studio's native panel-level features (per-panel SPL, per-panel drilldown, per-panel options UI) and have to use SPL or in-viz config to drive the grid. We believe a framework-level fix would unlock these use cases for every custom-viz app, not just ours.

We'd welcome a conversation with whoever owns the Dashboard Studio custom-viz runtime. Happy to share HAR captures, run additional reproducers, or test prototype fixes.
